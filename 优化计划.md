## 0. 当前 JWT 配置策略（避免「yml 删了还有冲突」的误解）

| 位置 | 现状 |
|------|------|
| `application.yml` | **已删除** `mes.jwt.*`、`mes.security.permit-urls` |
| `JwtUtil.java` | **硬编码** secret / issuer / 过期时间（当前真实生效处） |
| `SecurityConfig.java` | 白名单**写死在代码里**，不读 yml |

**结论**：删 yml 不会导致 Spring 启动报错（项目里没有任何 `@ConfigurationProperties` / `@Value` 绑定 `mes.jwt`）。  
若感觉「还有冲突」，通常是下面几种，**不是 yml 和 Spring 打架**：

1. **文档还写着读 yml**（`CLAUDE.md`、本文件以前的 Review 建议）→ 与「改在代码里配」不一致，以代码为准即可。  
2. **旧 Token 失效**：secret 或过期时间与之前不同，需重新登录。  
3. **Filter 与 Controller 响应格式不一致**：Filter 用 `sendError`，Controller 用 `Result` JSON。  
4. **计划文档**（`40天开发计划.md` D5）仍写「白名单读 `mes.security.permit-urls`」→ 需改成「SecurityConfig 硬编码」或以后自己实现读配置。

### 1.2 Authorization Header：`Bearer ` 前缀

- [ ] Filter 读取 token 时剥离 `mes` 或常量里的 `Bearer ` 前缀（若前端按标准传）  
- [ ] 与前端 / Apifox 约定统一格式  

### 1.3 Filter 内 LoginUser 占位字段

- [ ] `authorityList` 改为 `Collections.emptyList()`，不要 `null`  
- [ ] 缓存未接入前，`id` / `enabled` 不要传 `null`（避免 `isEnabled()` NPE）  
- [ ] 缓存接入后：从 Redis 还原**完整** `LoginUser` 再放入 `SecurityContext`  

### 1.4 错误响应格式统一

- [ ] Filter 鉴权失败目前 `response.sendError` → 与 `Result` JSON 不一致  
- [ ] 可选：自定义 `AuthenticationEntryPoint`，或 Filter 写 `{code,message,data}`  

### 1.5 LoginRequest 校验文案

- [ ] `@NotEmpty` 补 `message = "用户名不能为空"` 等，避免校验失败返回空 message

### 2.3 Security 白名单可维护性

- [ ] 现状：`SecurityConfig` 硬编码 `/api/auth/login` 等
